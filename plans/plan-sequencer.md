# Feature: File ID Sequencer

## 1. Purpose

The Sequencer generates globally unique, monotonically increasing `NeedleId` values used as the numeric part of every `FileId` assigned to a blob. It runs inside the Master Server and is the authoritative source of file IDs for the entire cluster.

Because every file upload needs a unique, never-reused ID, the sequencer must:
- Survive master restarts (IDs must not go backwards)
- Be consistent under leader failover (Raft-based option)
- Be performant (millions of IDs per second in batch mode)

## 2. Responsibilities

- Provide a `Next(count uint64)` call that returns the next `count` sequential NeedleIds
- Persist the current maximum issued ID to survive restarts without re-issuing IDs
- Support multiple backends: file-based (simple), Raft-replicated (HA), and snowflake (distributed without coordination)
- Allow the master to recover the max NeedleId seen in heartbeats from volume servers (for recovery after data loss)
- Support `SetMax(id)` to advance the counter when a higher ID is discovered

## 3. Non-Responsibilities

- Does not assign `Cookie` values (cookies are random, generated by the master at assign-time)
- Does not track which IDs are in use (that is the topology/volume manager's job)
- Does not generate `VolumeId` values (volumes are allocated by a separate counter)

## 4. Architecture Design

```
Master Server
    |
    +-- Sequencer (interface)
           |
    +------+------+
    |      |      |
  File   Raft  Snowflake
  Based  Based  Based

File-based:
  current_id stored in a single file: max_needle_id
  on Next(n): read, add n, write back, return old value + 1

Raft-based:
  current_id replicated in Raft log via MaxVolumeIdCommand
  on Next(n): Raft consensus write, return batch start

Snowflake-based (distributed, no coordination):
  id = timestamp_bits | instance_bits | sequence_bits
  no shared state required; IDs may not be strictly sequential
```

The **file-based** sequencer is the default for single-master mode. The **Raft-based** sequencer is used in multi-master HA mode. The **snowflake** sequencer is useful for write-heavy deployments where strict monotonic ordering is not required.

## 5. Core Data Structures (Go)

```go
package sequencer

import (
    "sync"
    "sync/atomic"
    "goblob/core/types"
)

// Sequencer is the interface for all sequencer implementations.
type Sequencer interface {
    // NextFileId returns the next batch of count globally unique NeedleIds.
    // The returned value is the start of the batch; the caller receives IDs
    // [start, start+count-1] inclusive.
    NextFileId(count uint64) uint64

    // SetMax advances the counter to at least maxId.
    // Used during recovery when heartbeats reveal a higher ID than stored.
    SetMax(maxId uint64)

    // GetMax returns the current maximum issued ID.
    GetMax() uint64
}

// FileSequencer is the default implementation for single-master mode.
// It stores the current max ID in a file on disk, batch-increments by
// a configurable step to amortize file I/O.
type FileSequencer struct {
    // dir is the directory where the max_needle_id file is stored.
    dir string
    // step is the pre-allocation batch size written to disk.
    // On crash, up to step IDs may be wasted. Default: 10000.
    step uint64
    // current is the last issued ID (in memory).
    current uint64
    // saved is the last value written to disk (current + step).
    saved uint64
    mu   sync.Mutex
}

// RaftSequencer uses Raft consensus to replicate ID assignments.
// It wraps a FileSequencer but commits increments via Raft log entries,
// ensuring all replicas have the same view of issued IDs.
type RaftSequencer struct {
    wrapped    *FileSequencer
    raft       RaftApplier  // interface to the Raft layer
    mu         sync.Mutex
}

// RaftApplier is the interface the sequencer uses to submit commands to Raft.
type RaftApplier interface {
    // Apply submits a command and waits for consensus.
    Apply(cmd []byte, timeout time.Duration) error
    // IsLeader reports whether this node is the Raft leader.
    IsLeader() bool
}

// SnowflakeSequencer generates IDs without coordination using a
// Twitter Snowflake-style scheme: timestamp + instance ID + sequence.
// IDs are globally unique but not guaranteed strictly sequential across instances.
type SnowflakeSequencer struct {
    instanceId uint64  // unique per master node (from IP/port hash)
    lastMs     int64   // last millisecond timestamp
    sequence   uint32  // per-millisecond counter
    mu         sync.Mutex
}

// Snowflake bit layout (64 bits total):
//   Bits 63:    0 (sign bit, always 0)
//   Bits 62-22: 41 bits timestamp (milliseconds since epoch, ~69 years)
//   Bits 21-12: 10 bits instance ID (up to 1024 instances)
//   Bits 11-0:  12 bits sequence (up to 4096 IDs per millisecond per instance)
const (
    snowflakeEpochMs  = int64(1700000000000) // arbitrary epoch: 2023-11-14
    snowflakeSeqBits  = 12
    snowflakeInstBits = 10
    snowflakeSeqMask  = (1 << snowflakeSeqBits) - 1
    snowflakeInstMask = (1 << snowflakeInstBits) - 1
)
```

## 7. Internal Algorithms

### FileSequencer.NextFileId(count)
```
mu.Lock()
defer mu.Unlock()

start = current + 1
current += count

if current > saved:
    saved = current + step
    writeToDisk(saved)   // sync write to max_needle_id file

return start
```

**Pre-allocation**: By writing `current + step` to disk instead of just `current`, the sequencer avoids a disk write on every call. After a crash, the next ID will start at `saved + 1`, wasting at most `step` IDs. This is safe because IDs are never reused.

**Disk format**: A single text file `max_needle_id` containing the decimal value of `saved`. Written atomically: write to `.tmp`, then `os.Rename`.

### FileSequencer.SetMax(maxId)
```
mu.Lock()
defer mu.Unlock()

if maxId > current:
    current = maxId
    if current > saved:
        saved = current + step
        writeToDisk(saved)
```

### RaftSequencer.NextFileId(count)
```
if !raft.IsLeader():
    return 0, ErrNotLeader   // caller should redirect to leader

mu.Lock()
defer mu.Unlock()

// Replicate via Raft before returning IDs
cmd = encodeMaxVolumeIdCommand(current + count + step)
raft.Apply(cmd, 5*time.Second)

start = current + 1
current += count
return start
```

The Raft command payload is `MaxVolumeIdCommand{MaxFileId: newMax}`. On commit, all Raft followers apply the command to advance their local `current`.

### SnowflakeSequencer.NextFileId(count)
```
// count is ignored; always returns 1 unique ID per call
mu.Lock()
defer mu.Unlock()

nowMs = unixMilliseconds() - snowflakeEpochMs
if nowMs == lastMs:
    sequence++
    if sequence > snowflakeSeqMask:
        // sequence exhausted this millisecond; busy-wait for next
        nowMs = waitNextMs(lastMs)
        sequence = 0
else:
    sequence = 0
    lastMs = nowMs

id = (nowMs << (snowflakeInstBits + snowflakeSeqBits)) |
     (instanceId << snowflakeSeqBits) |
     sequence
return uint64(id)
```

### SetMax on Recovery
During master startup (leader election), the master iterates all volume server heartbeats which contain the max NeedleId they have stored. It calls `seq.SetMax(maxNeedleId)` to ensure no ID is re-issued after a crash.

## 8. Persistence Model

### File Sequencer
```
$meta_dir/
  max_needle_id    # decimal text, e.g. "1000000"
  max_needle_id.tmp  # written then renamed atomically
```

### Raft Sequencer
The Raft log itself provides persistence. On follower recovery, the Raft log replay advances the counter to the correct value. The `FileSequencer` inside `RaftSequencer` serves as a local cache.

### Snowflake Sequencer
No persistence; state is derived from wall clock + instance ID. Safe because wall clock advances monotonically and instance IDs are unique.

## 9. Concurrency Model

All three implementations use `sync.Mutex` for thread safety. The `FileSequencer` holds the lock for the duration of the optional disk write (atomic rename is OS-level atomic but fast). The `RaftSequencer` holds the lock through the Raft `Apply` call (blocking until consensus reached, timeout ~5s).

The `SnowflakeSequencer` holds the lock only for the bit manipulation and clock read; no I/O under lock.

## 10. Configuration

```go
type SequencerConfig struct {
    // Type selects the sequencer backend: "file", "raft", "snowflake".
    Type string `mapstructure:"type"`
    // Dir is the directory for sequencer state files.
    Dir string `mapstructure:"dir"`
    // Step is the pre-allocation batch size for the file sequencer. Default: 10000.
    Step uint64 `mapstructure:"step"`
    // InstanceId is used by the snowflake sequencer. Derived from IP:port hash if not set.
    InstanceId uint64 `mapstructure:"instance_id"`
}
```

Default: `Type="file"` for single-master, `Type="raft"` when peers are configured.

## 11. Observability

- `obs.MasterAssignRequests.WithLabelValues("ok").Add(float64(count))` on each `NextFileId` call
- The current max ID is exposed via `expvar.Int` at `/debug/vars` key `"sequencer.max_id"`
- Disk write events logged at DEBUG level (too frequent for INFO)
- SetMax calls that advance the counter logged at INFO: `"sequencer advanced max id from X to Y"`

## 12. Testing Strategy

- **Unit tests**:
  - `TestFileSequencerMonotonic`: 1000 sequential calls, assert strictly increasing
  - `TestFileSequencerBatch`: call `Next(100)`, assert returns correct start, disk has `saved = start + 100 + step - 1`
  - `TestFileSequencerRestart`: write state, create new sequencer from same dir, assert continues from saved
  - `TestFileSequencerSetMax`: call `SetMax` with higher value, assert next `NextFileId` returns `maxId + 1`
  - `TestSnowflakeUniqueness`: generate 10000 IDs, assert no duplicates (use a map)
  - `TestSnowflakeMonotonic`: generate IDs for 100ms, assert sorted order within single goroutine
  - `TestRaftSequencerRedirects`: call `NextFileId` on a non-leader, assert `ErrNotLeader`
- **Concurrent tests**:
  - `TestFileSequencerConcurrent`: 100 goroutines each calling `Next(1)`, assert no duplicate IDs returned
